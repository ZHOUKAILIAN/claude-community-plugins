- **标题**: 图表优先的技术方案设计原则
- **版本**: 1.0.0
- **创建日期**: 2026-03-12
- **状态**: 草案

## 1. 需求背景

### 1.1 问题现状

目前的技术方案文档在阐述系统架构、数据工作流以及复杂交互时，过多依赖“纯文本叙述”与“大面积的列表”。这不仅导致用户（评审者、开发者、甚至解析文档的 AI Agent）阅读负担沉重、信息提取效率低下，而且由于自然语言的模糊性，极易在团队沟通中引发逻辑理解偏差。

### 1.2 目标用户

- 插件设计与开发人员
- 代码审查与技术评审人员
- 协作开发的 AI Agent 智能体

## 2. 功能需求

### 2.1 核心功能

**F1: 确立“图表与表格优先”的文档表达原则**
- 在所有后续产生的技术设计文档（`docs/design/`）中，架构设计、数据结构、业务工作流等内容，必须优先采用 Markdown Table 和 Mermaid 图表 进行展示。
- 只有在图表无法完整覆盖补充细节的前提下，才辅以少量、精简的文字说明。

**F2: 标准化核心设计节点的展示格式**
- **架构与时序**：强制使用 Mermaid（如 `flowchart`, `sequenceDiagram`, `classDiagram`）进行图形化呈现。
- **数据库与实体模型**：强制使用 Mermaid 的 ER 图（`erDiagram`）来表示数据库表结构及关联关系。
- **数据与参数**：强制使用 Markdown 表格（包含 字段、类型、必填项、说明 等列）替代无序/有序文本列表。

**F3: 标准化技术改动的差异展示（Inline Diff）**
- **同位对比**：改动前（Before）和改动后（After）的差异必须在同一个图表或同一个表格中直观展示，禁止强行拆分为“改前”和“改后”两章。
- **视觉高亮**：
  - 在 Markdown 表格中，使用 `~~删除线~~` 或 HTML 标签（如 `<del>`, `<ins>` 或自带颜色的 `<span style="color:red">`）标出字段的增删改。
  - 在 Mermaid 图表中，利用其自带的 `style` 或 `classDef` 语法给新增的节点标绿、删除的节点标红、修改的节点标黄，从而在一张图内全景展示架构演进。

### 2.2 辅助功能

- 更新项目的文档模板结构，以示例的形式指导新进开发者（及 AI）如何按照“图表优先”的思想编写文档。

## 3. 技术需求

### 3.1 架构设计

无具体代码架构变更，核心为开发标准库（Standard Library / Templates）的改版设计。预期将引入新的规范段落和图表示例占位符至技术方案模板中。

### 3.2 技术实现大纲

1. 审查和更新现有的文档规范模板：
   - `docs/standards/technical-design-template.md` （核心）
   - `docs/standards/requirements-template.md` （可选扩展）
2. 在模板中植入显式的 `<!-- 强约束：必须使用 Mermaid 图表展示 -->` 等指引注释。

### 3.3 分项目类型的详细规范（可选）

#### 3.3.1 文档生成规范
- AI Agent 在基于此需求进行文档生成任务时，对于任何带有分支逻辑的工作流，默认需将其转换为 Mermaid 的条件判断图形式。

## 技术栈

- **Markdown** (GFM: 扩展的表格语法的支持)
- **Mermaid.js** (流程图, 时序图, 架构图等)

## 开发工作流程

### 1. 强制性文档优先原则

在本次和后续的迭代中，继续遵循完全的“AI 辅助文档驱动开发”规范，新需求全部落实到 `REQ-YYYYMMDD` 和 `TECH-YYYYMMDD`，由用户批准后再执行代码（模板）层面的改写。

### 2. 开发步骤（严格按顺序执行）

1. 创建并确立本需求文档与对应的技术设计文档。
2. 提交用户审查确认方向。
3. 执行技术设计文档中规划的对 `docs/standards/*.md` 模板的内容改写。

## 4. 风险评估

### 4.1 技术风险

- **Markdown 渲染差异**：部分较为陈旧的 Markdown 阅读器或特定平台可能存在对过于复杂的 Mermaid 图表语法不支持的情况。但鉴于目标平台主要为 GitHub 与 AI 原生面板，该风险极低，优先选用标准且基础的 `flowchart TD` 与 `sequenceDiagram`。
- **编写成本变高**：相较于信马由缰地书写文字，设计严谨的图表会占用初期编写时间。但这将在后期的代码维护和团队沟通中得到数倍的时间回报。
